## Problemas Relatados pelo Usuário

### 1. Abacate Pay — API V2 não funciona
- Código tentava V2 primeiro mas sempre caía em fallback pra V1 com erro 401
- V2 é de subscriptions, V1 é billing one-time
- Usuário queria remover V2 completamente

### 2. Cupom de teste grátis
- Cupom antigo: `TRIAL7DIAS` (não funcionava)
- Cupom novo criado no dashboard: `FREE TEST` (100% desconto)
- Teste grátis deve durar 7 dias, único por conta

### 3. Dados pessoais vazando entre usuários (CRÍTICO)
- **Sintoma:** Ao criar conta nova e fazer checkout, os dados pessoais (nome, email) do ELIAS apareciam no checkout da NOVA conta
- **Causa raiz:** A função `createCustomerIfMissing` buscava customer existente por email/CPF no AbacatePay → encontrava o ÚNICO customer no painel (do Elias) → TODOS os checkouts usavam o MESMO customer
- **Impacto:** Todos os usuários viam os dados do Elias no checkout

### 4. Desconto do trial não aplicava
- Mesmo com `isTrial: true` no payload, o cupom não era aplicado
- Valor continuava R$ 19,90 em vez de R$ 0,00

### 5. Cancelamento de assinatura inexistente
- Não havia botão/função para cancelar assinatura
- Texto "Cancelamento e retomada pela conta" era apenas marketing, sem implementação

### 6. Persistência de perfil entre contas
- Ao deslogar do "Elias Nunes" e logar no "Elias Juriatti", o nome/email do app continuava no Elias Nunes
- **Nota:** Isso é SEPARADO do cloud sync — o perfil vem do JWT do Netlify Identity, não do localStorage/blobs

### 7. Efeitos visuais com bugs
- Gradiente aparecendo em cards quando toggle estava desligado
- Navbar NÃO tinha gradiente quando toggle estava ligado (deveria ter)
- Botões de seleção de tema da redação tinham gradiente indevido

### 8. Cloud Sync — sobrescrevia dados locais
- `applyAccountSnapshot` sobrescrevia tudo que vinha da nuvem
- Dados locais eram perdidos no pull
- Aparência (tema, fonte) podia ser sobrescrita

### 9. JWT inchado (problema histórico)
- `user_metadata` continha `apice_state` pesado → JWT com 500+ chars → erro 500 no API Gateway
- Solução paliativa: `X-User-Id` header (forjável — qualquer um pode enviar ID arbitrário)

## Próximos Passos — Fase 1: JWT Seguro

> [!IMPORTANT]
> **SÓ PROSSEGUIR APÓS:**
> - ✅ Compra/checkout funcionar (trial aplica desconto, dados não vazam)
> - ✅ Cloud sync funcionar (pull/pull correto, merge inteligente)
> - ✅ Perfil persistindo entre contas RESOLVIDO

### Problema
O `X-User-Id` é trivialmente forjável — qualquer pessoa pode enviar um ID arbitrário e acessar dados alheios. O JWT original dava 500 porque o `user_metadata` continha `apice_state` pesado.

### Solução
O `apice_state` foi removido do `user_metadata` (migrado para Blobs). O JWT agora é leve o suficiente para ser usado como auth primária.

### Estratégia

1. **Frontend (`authFetch.js`):**
   - Enviar `Authorization: Bearer <jwt>` em vez de `X-User-Id`
   - `registerAuthTokenGetter` retorna JWT via `currentUser().jwt()`
   - Remover fallback para `localStorage.getItem('gotrue.user')`
   - Manter X-User-Id como fallback secundário (período de transição)

2. **Backend (`auth.js`):**
   - `requireAuth()`: Priorizar JWT (`Bearer` token) sobre `X-User-Id`
   - Manter `X-User-Id` como fallback com log de warning
   - Decodificação estrutural (Base64 + JSON parse) — sem verificação criptográfica (aceitável para Netlify Identity)

3. **AuthProvider:**
   - Registrar JWT getter via `currentUser().jwt()` em vez de `currentUser().id`
   - No `logout()`: garantir limpeza completa do JWT

4. **CORS (`cors.js`):**
   - Adicionar `X-User-Id` ao `Access-Control-Allow-Headers` (transição)

### Arquivos a Modificar

| Arquivo | Mudança |
|---------|---------|
| `src/services/authFetch.js` | Enviar Bearer token, registrar JWT getter |
| `src/auth/AuthProvider.jsx` | Usar `currentUser().jwt()`, preservar aparência no logout |
| `netlify/functions/utils/auth.js` | Priorizar JWT sobre X-User-Id |
| `netlify/functions/utils/cors.js` | Headers atualizados |
